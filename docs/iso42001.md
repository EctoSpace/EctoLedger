## 1. Executive Summary

Ecto Ledger is a cryptographically-verifiable, hash-chained event log for autonomous AI agents. This whitepaper demonstrates how the Ecto Ledger system satisfies the requirements of **ISO/IEC 42001:2023** — the first international standard specifying requirements for an *AI Management System* (AIMS).

The ledger provides:

- **Immutability** — SHA-256 hash-chaining ensures no event can be silently altered.
- **Non-repudiation** — Ed25519 per-session signing keys provide cryptographic proof of authorship.
- **Observability** — Policy-driven observation rules, Prometheus metrics, and SSE event streams give real-time situational awareness.
- **Governance** — Formal logic policies (SOC2, PCI-DSS, OWASP, ISO 42001) are loaded at runtime, version-pinned, and hash-recorded per session.
- **Accountability** — Every agent action, guard decision, and tripwire rejection is appended to the ledger and linked to the original session goal via `goal_hash`.

## 4. Cryptographic Controls

### 4.1 Hash-Chaining

Every event record contains:
```
content_hash = SHA-256(previous_content_hash || sequence || JSON(payload))
```
This links events in a tamper-evident chain identical in structure to append-only logs used in blockchain and supply-chain provenance systems.

### 4.2 Event Signing

Sessions are assigned a fresh Ed25519 keypair on creation. The public key is recorded in `agent_sessions.session_public_key`. Each event's `content_hash` is signed and stored in `agent_event_signatures`, enabling offline verification:

```bash
ectoledger verify-cert --session <UUID>
```

### 4.3 Decentralized Identifiers

In `enterprise-2026`, a `session_did` string is stored alongside the session key. Phase 1 uses `did:key:` derived from the Ed25519 public key (per the W3C DID Key specification), providing linkability between sessions and operator identities without a centralised registry.

### 4.4 Merkle Snapshots

At configurable intervals the system computes a Merkle root over all event content hashes up to the current sequence. Stored in `agent_snapshots`, this root allows O(log n) proof that any single event is included, enabling efficient third-party audits without replaying the full history.
